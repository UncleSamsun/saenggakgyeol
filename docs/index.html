Development plan

구현은 아래 순서로 진행합니다. 각 단계가 끝날 때마다 이 페이지의 상태와 artifact 링크를 갱신합니다.

Phase 1: Capture skeleton
메뉴바 앱, 전역 단축키, floating session editor, timestamped notes. 첫 검증은 “30분짜리 긴 notes session을 저장할 수 있는가”입니다.
Phase 2: Obsidian output + local index
Markdown writer, SQLite session index, processing state. Markdown은 source of truth, DB는 cache/index입니다.
Phase 3: Recording + transcript
녹음 토글, 권한 흐름, transcript chunk 저장, note chunk와 시간 매칭.
Phase 4: AI synthesis
End Session 시 summary, decisions, tasks, questions, evidence snippets를 생성합니다. 실패 상태도 UI에 보여야 합니다.
Phase 5: Widget + release polish
Mac widget, demo screenshots, GitHub release, README polish, portfolio story.

Repository structure

Pages site는 docs/를 정적 사이트 루트로 사용합니다. GitHub Actions가 이 폴더를 Pages artifact로 업로드합니다. 

.
├── README.md
├── .github/
│   └── workflows/
│       └── deploy-pages.yml
└── docs/
    ├── .nojekyll
    ├── README.md
    ├── index.html              # GitHub Pages main tracker
    └── product/
        └── index.html          # Product concept detail

Update rules

  • 새 기획 문서가 생기면 Documents에 링크를 추가한다.
  • 새 개발 단계가 시작되면 Master tracker의 status를 바꾼다.
  • 커다란 제품 결정은 Decision timeline에 한 줄로 남긴다.
  • 각 phase가 끝나면 산출물 링크와 검증 결과를 메인 페이지에 추가한다.
  • Pages 배포가 실패하면 workflow 로그와 수정 커밋을 tracker에 남긴다.
